# 前端使用者體驗檢視報告

**檢視日期**：2026-06-23
**檢視對象**：`frontend/`（index.html / src/main.ts / src/viewer.ts / src/api.ts / src/style.css）
**依據**：`frontend_review_tree.md`（10 層使用者體驗檢查清單）

**總評**：**核心流程已合格可用（功能與資料流正確、錯誤處理完整），但「打磨層」尚未達標**——
缺乏載入進度、無障礙標註、響應式（行動裝置）、3D 視口空狀態引導等。
適合內部 / 開發者使用；若要對一般使用者公開發布，建議補完下方標 ❌ 的項目。

評分圖例：✅ 合格　🟡 部分合格／可改善　❌ 不合格／缺失

| 項目 | 狀態 | 說明 |
|------|------|------|
| 7.1 鍵盤可操作 | 🟡 | 原生 input/button 可 Tab 聚焦；但 3D 視口無鍵盤旋轉替代 |
| 7.2 焦點樣式可辨識 | 🟡 | 依賴瀏覽器預設 focus ring，未自訂；深色底下可能不明顯 |

**建議（高優先）**：狀態列加 `role="status" aria-live="polite"`；自訂 focus 樣式；滑桿補 `aria-*`。

## 9. 效能與流暢層 — 🟡

| 項目 | 狀態 | 說明 |
|------|------|------|
| 9.1 首屏夠快 | 🟡 | Vite 產物小，但 Three.js 主 chunk 588KB，首載略重 |
| 9.2 互動不卡頓 | ✅ | OrbitControls + `enableDamping`，旋轉順暢；`requestAnimationFrame` 迴圈正確 |
| 9.3 無不必要重渲染 | 🟡 | 每幀都 render（3D 檢視器常見作法）；靜止時仍持續繪製，可用 `controls.addEventListener('change')` 按需渲染省電 |
| 9.4 資源大小受控 | 🟡 | 後端 .glb 為頂點級網格（H×W 頂點），大圖會產生大網格；前端未做面數上限提示 |
| 9.5 大量資料漸進顯示 | 🟡 | .glb 一次載入；中等圖可接受，超大圖無串流 |

**建議**：靜止時停止重繪（按需渲染）；對超大輸入給面數警示。

## 總結與優先建議

**目前合格度**：核心可用、資料流與錯誤處理正確、視覺一致 ✅；
但「對一般使用者公開發布」所需的打磨尚缺，集中在 **第 6（響應式）、第 7（無障礙）、第 8.1（空狀態）、第 2.2（載入進度）**。

**建議補完順序（投入小、效益大）**：
1. **載入進度回饋**（2.2 / 8.2）——合成中加 spinner / 進度提示，消除「是否當機」疑慮。
2. **3D 視口空狀態**（8.1 / 1.5）——初始顯示「上傳並合成後，3D 照片會出現在此」。
3. **無障礙基礎**（7.4 / 7.2）——狀態列 `aria-live`、自訂 focus 樣式、滑桿 `aria-*`。
4. **響應式版面**（6.1 / 5.6）——`@media (max-width:768px)` 面板改上下堆疊。
5. **拖放上傳 + 範例檔**（4.4）——降低首次使用門檻。
6. **按需渲染**（9.3）——靜止不重繪，省電省效能。

> 1～3 屬「低成本高回報」，建議優先；4～6 視是否面向一般大眾發布而定。
> 這些皆為前端 `frontend/` 範圍內的調整，不影響後端與核心管線。
